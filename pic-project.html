Data Parallelization on GPU Cluster

C · GPU Cluster · Image Processing · Data Parallelism

Problem

Applying image filters to large photos on a single node is slow. A convolution kernel touches every pixel and its neighborhood, so the work is embarrassingly parallel but needs correct border handling, a clean filter interface, and a way to spread rows across cluster GPUs.

Solution

I built a data-parallel convolution pipeline that runs 3×3 kernel filters across a GPU cluster. Given an input file and a filter type, the program loads the image, partitions rows across workers, convolves each pixel against a kernel matrix, and writes the result as PNG. Six filters are supported: edge, sharpen, blur, Gaussian blur, emboss, and identity.

Each output pixel is a weighted sum of its 3×3 neighborhood. Edge pixels clamp to the nearest border value so the kernel never reads out of bounds. Row batches are distributed across cluster nodes so each GPU processes a slice of the image in parallel.

The same getPixelValue function drives every filter on each worker; only the kernel matrix changes between modes. Results are gathered back into a single output image once all partitions finish. 

// edge detection kernel
{ {0,-1,0}, {-1,4,-1}, {0,-1,0} }

// sharpen kernel
{ {0,-1,0}, {-1,5,-1}, {0,-1,0} }

// blur, gaussian, emboss, and identity kernels follow the same layout

Outcome

The same pipeline produces six filter outputs from one input image, with data parallel execution spread across the GPU cluster. Filter selection is a single kernel matrix swap with no duplicated convolution logic.
