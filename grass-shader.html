Custom Grass Shader Tool

Unity · C# · Artist Tools · HLSL

Problem

Procedural grass in Unity usually means editing compute shader constants or recompiling HLSL every time an artist wants to change density, blade height, wind strength, or LOD behavior. That slows iteration and keeps shader knowledge on the engineering side of the team.

Solution

I built a tool for customizing a procedural grass shader: an in-editor inspector UI with exposed parameters that map directly to grass and wind settings on ProceduralGrassRenderer. Artists can iterate in play mode without touching code. The tool can also generate a tileable Perlin noise texture for wind sampling.

Under the hood, a compute shader generates blade geometry on the GPU (one thread per source triangle), streams output to an append buffer, and draws in a single indirect call. The editor tool is the interface; the shader system is the engine behind it.

Compute shader details

Each thread handles one triangle from the source mesh, finds the triangle center in world space, builds a tangent-to-world matrix, and sprouts a blade from that point. Blade width and height are randomized per position. A blade is a strip of square sections stacked along height, with segment count reduced at distance via LOD.

Bend, twist, and wind rotation apply in order. Wind strength comes from procedural gradient noise sampled in world space and scrolled over time, so the base stays anchored and the tip moves the most.

Outcome

Artists can customize the full grass field from the inspector and see results immediately. Shader complexity stays encapsulated behind exposed parameters and sensible defaults.
