Editable Blender Shader Extension

Blender · OpenUSD · UsdPreviewSurface · Procedural Materials

Problem

A weathered metallic surface with water droplets is difficult to paint by hand, and one-off shader node graphs do not travel well into a pipeline. Artists need editable parameters, and downstream tools need a material that can export to interchange formats like OpenUSD through baked PBR textures and UsdPreviewSurface.

Solution

I built a custom editable shader extension in Blender: a reusable procedural material assembled layer by layer in the node editor, with exposed inputs for scale, roughness, and droplet density. The graph uses Musgrave multifractal for base variation, distorted Voronoi for droplet regions, normal mapping for surface relief, and a metallic BSDF for the final specular response.

The node group is structured for pipeline handoff: procedural layers can be baked to base color, roughness, metallic, and normal maps, then assigned through Principled BSDF and exported as OpenUSD UsdPreviewSurface.

I started with a Musgrave texture set to multifractal mode, mapped onto the sphere's surface coordinates. This gave a base layer of organic variation with streaky irregular bright spots against a dark field.

On top of that I layered a Voronoi texture, distorted with a Mapping node and Vector Math to break up the regular cell pattern into soft, blob-like droplet regions.

Feeding the combined noise into a Normal Map node gave the first bump pass. The droplets read as small raised spots, but the scale was too fine and the shapes too uniform.

Adjusting the Musgrave scale and dimension settings pushed the bumps larger and more teardrop-shaped, as if water is pooling and running down the surface.

OpenUSD export pipeline

Procedural nodes do not export directly. The reliable path is to bake the procedural layers to image textures, plug those into Principled BSDF, and let Blender's USD exporter convert Principled BSDF to UsdPreviewSurface. The procedural node group stays in the file (disabled, not deleted) as the editable source; the baked material is the export copy.

1. UV setup

The mesh needs a proper UV unwrap because baking writes to UV space. For the sphere, Smart UV Project or Unwrap in edit mode is enough.

2. Bake target images

In the Shader Editor, create one Image Texture node per map: base color, roughness, metallic, and normal. Name them clearly (metal_basecolor, metal_roughness, metal_metallic, metal_normal). 2048×2048 is a reasonable starting size. Set the normal map image to Non-Color; base color stays sRGB. Select the target Image Texture node before each bake so Cycles writes to the right image.

3. Bake in Cycles

Switch to Cycles (Render Properties → Render Engine → Cycles). In Render Properties → Bake:

Base color: Bake Type = Diffuse, uncheck Direct and Indirect, keep only Color (albedo without lighting baked in).
Roughness: Bake Type = Roughness.
Metallic: No built-in Metallic bake type. Temporarily connect the metallic factor to an Emission shader's color input and bake as Emit, then restore the original graph.
Normal: Bake Type = Normal, Space = Tangent.

4. Normal map range on export

Cycles stores tangent-space normals as a standard 0-1 image (128, 128, 255 = flat), which is correct for the file. The common failure is in the USD exporter feeding that 0-1 texture into UsdPreviewSurface's normal input without remapping to the signed -1 to 1 range. If normals look skewed or inverted in Houdini or usdview after export, open the .usda and check the normal UsdUVTexture node: it should have scale (2, 2, 2, 1) and bias (-1, -1, -1, 0). Add them manually if missing, or confirm your Blender version includes the fix in its changelog.

5. Rebuild with baked textures

Add Image Texture nodes for each baked map and plug them into a fresh Principled BSDF: base color → Base Color, roughness → Roughness, metallic → Metallic, normal → Normal Map node → Normal input. The original procedural group remains available for iteration; this Principled setup is what gets exported.

6. Export and verify

File → Export → Universal Scene Description (.usd / .usda / .usdc) with Materials enabled. Use .usda first so you can read the shader network and confirm UsdPreviewSurface with UsdUVTexture nodes feeding it. Round-trip in usdview or Blender's USD importer in a fresh scene to verify the look outside the original file.

Outcome

The procedural extension stays editable in Blender, and the baked Principled material exports through OpenUSD as UsdPreviewSurface with standard PBR texture inputs. Pipeline handoff becomes a concrete .usda asset rather than a Blender-only node graph.
