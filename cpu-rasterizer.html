CPU Rasterizer

C++ · Software Rendering · Computer Graphics

Problem

GPU rendering hides the pipeline behind driver APIs. To understand how meshes become pixels, you need a renderer where every stage is visible: rasterization, depth testing, attribute interpolation, and texture sampling.

Solution

A software rasterizer built from scratch on the CPU, following Tayfun Kayhan's Rasterization in One Weekend series. The repo grows in three stages: a flat triangle, shaded cubes in perspective, then a textured Sponza atrium loaded from disk. Each stage reuses the same edge function core; later stages add depth testing, backface culling, and perspective correct attribute interpolation.

Part I: Hello triangle

The first milestone rasterizes a single triangle in 2D. Vertices start in normalized device coordinates (NDC), then a TO_RASTER macro maps them into pixel space so every test happens in the same coordinate system: 

#define TO_RASTER(v) glm::vec3(
  (g_scWidth  * (v.x + 1.0f) / 2),
  (g_scHeight * (v.y + 1.0f) / 2),
  1.0f)

To decide whether a pixel center falls inside the triangle, the rasterizer uses edge functions. Each edge defines a half-space; if a sample is on the same side of all three edges, it is inside. Barycentric coordinates are the weights that blend the three vertex colors. They are a linear combination of the three vertices that sums to 1: a point closer to vertex 0 gets a larger alpha, closer to vertex 1 gets a larger beta, and so on.

The code builds a vertex matrix M from the three raster-space positions, inverts it, and reads off the edge vectors. At each pixel, alpha, beta, and gamma are dot products of those edges with the sample point; when all three are non-negative, the pixel is shaded by blending red, green, and blue. 

float alpha = glm::dot(E0, sample);
float beta  = glm::dot(E1, sample);
float gamma = glm::dot(E2, sample);

if ((alpha >= 0.0f) && (beta >= 0.0f) && (gamma >= 0.0f)) {
  frameBuffer[x + y * g_scWidth] = glm::vec3(c0 * alpha + c1 * beta + c2 * gamma);
}

Part II: Shaded cubes

The second stage moves into 3D. A cube is stored as a small set of unique vertices plus an index buffer, because adjacent triangles share corners and repeating every position would waste memory. Each cube instance gets its own model matrix; a shared view and projection matrix place the camera above the scene looking down.

A minimal vertex shader transforms object-space positions to clip space with P * V * M. Before homogeneous division, vertices map to raster space using the w component so perspective is preserved: 

#define TO_RASTER(v) glm::vec4(
  (g_scWidth  * (v.x + v.w) / 2),
  (g_scHeight * (v.w - v.y) / 2),
  v.z, v.w)

The vertex matrix now includes each corner's w value in the third row. Its determinant tells you whether a triangle faces the camera: if det(M) >= 0, the triangle is back-facing and is skipped. This is backface culling without a separate normal pass.

A depth buffer resolves overlap between cubes. Instead of storing raw z, the rasterizer interpolates 1/w across the triangle (perspective-correct depth) and keeps the fragment with the largest value, so nearer surfaces win. Shared edges between adjacent triangles use tie-breaking rules in EvaluateEdgeFunction so the same pixel is not shaded twice. 

float oneOverW = (C.x * sample.x) + (C.y * sample.y) + C.z;

if (oneOverW >= depthBuffer[x + y * g_scWidth]) {
  depthBuffer[x + y * g_scWidth] = oneOverW;
  frameBuffer[x + y * g_scWidth] = colors[indices[3 * idx] % 6];
}

Part III: Textured Sponza

The final stage loads a real mesh. tinyobjloader parses the OBJ and MTL files; stb_image loads each material's diffuse texture. Because OBJ indices can refer to the same position with different UV or normal indices, the loader builds an IndexedPrimitive key for each unique (position, normal, UV) triple and deduplicates vertices before filling a global index buffer.

Rendering splits into a vertex shader and fragment shader, mirroring a GPU pipeline. The VS applies the MVP transform and passes normals and texture coordinates through to the FS. The FS samples the diffuse map with repeat wrapping on fractional UV coordinates. 

glm::vec4 VS(const VertexInput& input, const glm::mat4& MVP, FragmentInput& output) {
  output.Normal    = input.Normal;
  output.TexCoords = input.TexCoords;
  return (MVP * glm::vec4(input.Pos, 1.0f));
}

Flat UV interpolation would smear textures on large triangles in perspective. The fix is to interpolate u/w and v/w at each vertex, then divide by the interpolated 1/w at the fragment. The same edge-function machinery also interpolates z for the depth test and normals if you switch the FS to visualize them. 

float uOverW = (PUVS.x * sample.x) + (PUVS.y * sample.y) + PUVS.z;
float vOverW = (PUVT.x * sample.x) + (PUVT.y * sample.y) + PUVT.z;
glm::vec2 texCoords = glm::vec2(uOverW, vOverW) / perspDenom;

Sponza exports vary widely in scale depending on the source file. Hardcoded eye and look-at values that work for one version produce a black frame or a close-up of a wall on another. The renderer walks all loaded vertices once to compute an axis-aligned bounding box, then derives camera position, look target, and near/far planes from the box center, size, and diagonal. It picks the longer horizontal axis so the view looks down the length of the atrium rather than into a side wall.

One practical note from the code: if Sponza renders completely black, the first thing to check is winding order. Some OBJ exports use the opposite face orientation, which makes the backface test cull every triangle. Flipping the det(M) comparison confirms whether that is the issue.

Outcome

The renderer produces correct PPM output from hello triangle through a fully textured architectural scene, with a scale invariant camera derived from model bounds. Switch between the three demos in main.cpp by uncommenting HelloTriangle, Cube, or GoWild.
